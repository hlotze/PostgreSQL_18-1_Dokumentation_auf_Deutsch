# 57. Einen Handler für prozedurale Sprachen schreiben

Alle Aufrufe von Funktionen, die in einer Sprache geschrieben sind, die nicht die aktuelle »Version-1«-Schnittstelle für kompilierte Sprachen verwendet, laufen über eine sprachspezifische Call-Handler-Funktion. Dazu gehören Funktionen in benutzerdefinierten prozeduralen Sprachen ebenso wie in SQL geschriebene Funktionen. Der Call Handler ist dafür verantwortlich, die Funktion sinnvoll auszuführen, etwa indem er den übergebenen Quelltext interpretiert. Dieses Kapitel skizziert, wie ein neuer Call Handler für eine prozedurale Sprache geschrieben werden kann.

Der Call Handler einer prozeduralen Sprache ist eine »normale« Funktion, die in einer kompilierten Sprache wie C geschrieben sein muss, die Version-1-Schnittstelle verwendet und in PostgreSQL als Funktion ohne Argumente mit Rückgabetyp `language_handler` registriert wird. Dieser spezielle Pseudotyp kennzeichnet die Funktion als Call Handler und verhindert, dass sie direkt in SQL-Befehlen aufgerufen wird. Weitere Einzelheiten zu C-Aufrufkonventionen und dynamischem Laden finden Sie in [Abschnitt 36.10](36_SQL_erweitern.md#3610-csprachfunktionen).

Der Call Handler wird wie jede andere Funktion aufgerufen: Er erhält einen Zeiger auf eine `FunctionCallInfoBaseData`-Struktur, die Argumentwerte und Informationen über die aufgerufene Funktion enthält, und soll ein `Datum`-Ergebnis zurückgeben. Wenn ein SQL-Nullwert zurückgegeben werden soll, kann er außerdem das Feld `isnull` der `FunctionCallInfoBaseData`-Struktur setzen. Der Unterschied zwischen einem Call Handler und einer gewöhnlichen aufgerufenen Funktion besteht darin, dass das Feld `flinfo->fn_oid` der `FunctionCallInfoBaseData`-Struktur die OID der tatsächlich aufzurufenden Funktion enthält, nicht die des Call Handlers selbst. Der Call Handler muss dieses Feld verwenden, um zu bestimmen, welche Funktion auszuführen ist. Außerdem wurde die übergebene Argumentliste gemäß der Deklaration der Zielfunktion aufgebaut, nicht gemäß der Deklaration des Call Handlers.

Der Call Handler muss den Eintrag der Funktion aus dem Systemkatalog `pg_proc` holen und die Argument- und Rückgabetypen der aufgerufenen Funktion analysieren. Die `AS`-Klausel aus dem `CREATE FUNCTION`-Befehl der Funktion befindet sich in der Spalte `prosrc` der `pg_proc`-Zeile. Üblicherweise ist das Quelltext in der prozeduralen Sprache; theoretisch könnte es aber auch etwas anderes sein, zum Beispiel ein Pfadname zu einer Datei oder jede andere Angabe, die dem Call Handler mitteilt, was im Einzelnen zu tun ist.

Häufig wird dieselbe Funktion innerhalb einer SQL-Anweisung mehrfach aufgerufen. Ein Call Handler kann wiederholte Nachschlagevorgänge zu Informationen über die aufgerufene Funktion vermeiden, indem er das Feld `flinfo->fn_extra` verwendet. Dieses Feld ist anfangs `NULL`, kann aber vom Call Handler so gesetzt werden, dass es auf Informationen über die aufgerufene Funktion zeigt. Bei späteren Aufrufen kann der Handler, falls `flinfo->fn_extra` bereits nicht `NULL` ist, diese Informationen verwenden und das erneute Nachschlagen überspringen. Der Call Handler muss sicherstellen, dass `flinfo->fn_extra` auf Speicher zeigt, der mindestens bis zum Ende der aktuellen Abfrage gültig bleibt, da eine `FmgrInfo`-Datenstruktur so lange aufbewahrt werden kann. Eine Möglichkeit ist, die Zusatzdaten im durch `flinfo->fn_mcxt` angegebenen Speicher-Kontext zu allozieren; solche Daten haben normalerweise dieselbe Lebensdauer wie die `FmgrInfo` selbst. Der Handler kann sich aber auch für einen langlebigeren Speicher-Kontext entscheiden, um Informationen über Funktionsdefinitionen abfrageübergreifend zwischenzuspeichern.

Wenn eine Funktion einer prozeduralen Sprache als Trigger aufgerufen wird, werden keine Argumente auf die übliche Weise übergeben. Stattdessen zeigt das Feld `context` von `FunctionCallInfoBaseData` auf eine `TriggerData`-Struktur, anstatt wie bei einem einfachen Funktionsaufruf `NULL` zu sein. Ein Language Handler sollte Mechanismen bereitstellen, mit denen Funktionen der prozeduralen Sprache auf die Trigger-Informationen zugreifen können.

Eine Vorlage für einen als C-Erweiterung geschriebenen Handler für prozedurale Sprachen befindet sich in `src/test/modules/plsample`. Dies ist ein funktionsfähiges Beispiel, das eine Möglichkeit zeigt, einen Handler für prozedurale Sprachen zu erzeugen, Parameter zu verarbeiten und einen Wert zurückzugeben.

Ein Call Handler reicht zwar aus, um eine minimale prozedurale Sprache zu erstellen, es können aber optional zwei weitere Funktionen bereitgestellt werden, um die Sprache bequemer nutzbar zu machen: ein Validator und ein Inline Handler. Ein Validator erlaubt sprachspezifische Prüfungen während `CREATE FUNCTION`. Ein Inline Handler erlaubt der Sprache, anonyme Codeblöcke zu unterstützen, die mit dem Befehl `DO` ausgeführt werden.

Wenn eine prozedurale Sprache einen Validator bereitstellt, muss dieser als Funktion mit einem einzelnen Parameter vom Typ `oid` deklariert werden. Das Ergebnis des Validators wird ignoriert, daher wird er üblicherweise mit Rückgabetyp `void` deklariert. Der Validator wird am Ende eines `CREATE FUNCTION`-Befehls aufgerufen, der eine in der prozeduralen Sprache geschriebene Funktion erzeugt oder aktualisiert hat. Die übergebene OID ist die OID der `pg_proc`-Zeile der Funktion. Der Validator muss diese Zeile auf die übliche Weise holen und die jeweils passenden Prüfungen durchführen.

Zuerst sollte `CheckFunctionValidatorAccess()` aufgerufen werden, um explizite Aufrufe des Validators zu diagnostizieren, die der Benutzer nicht über `CREATE FUNCTION` erreichen könnte. Typische Prüfungen stellen anschließend sicher, dass die Argument- und Ergebnistypen der Funktion von der Sprache unterstützt werden und dass der Funktionskörper syntaktisch korrekt ist. Wenn der Validator die Funktion für in Ordnung befindet, sollte er einfach zurückkehren. Findet er einen Fehler, sollte er ihn über den normalen `ereport()`-Mechanismus melden. Das Werfen eines Fehlers erzwingt einen Transaktions-Rollback und verhindert so, dass die fehlerhafte Funktionsdefinition festgeschrieben wird.

Validatorfunktionen sollten normalerweise den Parameter `check_function_bodies` beachten: Ist er ausgeschaltet, sollten teure oder kontextabhängige Prüfungen übersprungen werden. Wenn die Sprache Codeausführung zur Kompilierzeit vorsieht, muss der Validator Prüfungen unterdrücken, die eine solche Ausführung auslösen würden. Insbesondere schaltet `pg_dump` diesen Parameter aus, damit prozedurale Sprachfunktionen geladen werden können, ohne sich um Seiteneffekte oder Abhängigkeiten der Funktionskörper von anderen Datenbankobjekten kümmern zu müssen. Wegen dieser Anforderung sollte der Call Handler nicht annehmen, dass der Validator die Funktion vollständig geprüft hat. Der Zweck eines Validators besteht nicht darin, dem Call Handler Prüfungen zu ersparen, sondern den Benutzer sofort auf offensichtliche Fehler in einem `CREATE FUNCTION`-Befehl hinzuweisen.

Welche Prüfungen genau vorgenommen werden, bleibt weitgehend der Validatorfunktion überlassen. Beachten Sie aber, dass der Kerncode von `CREATE FUNCTION` an eine Funktion angehängte `SET`-Klauseln nur ausführt, wenn `check_function_bodies` eingeschaltet ist. Prüfungen, deren Ergebnis von GUC-Parametern beeinflusst werden könnte, sollten daher unbedingt übersprungen werden, wenn `check_function_bodies` ausgeschaltet ist, um falsche Fehler beim Wiederherstellen eines Dumps zu vermeiden.

Wenn eine prozedurale Sprache einen Inline Handler bereitstellt, muss dieser als Funktion mit einem einzelnen Parameter vom Typ `internal` deklariert werden. Das Ergebnis des Inline Handlers wird ignoriert, daher wird er üblicherweise mit Rückgabetyp `void` deklariert. Der Inline Handler wird aufgerufen, wenn eine `DO`-Anweisung mit der prozeduralen Sprache ausgeführt wird. Der tatsächlich übergebene Parameter ist ein Zeiger auf eine `InlineCodeBlock`-Struktur, die Informationen über die Parameter der `DO`-Anweisung enthält, insbesondere den Text des auszuführenden anonymen Codeblocks. Der Inline Handler sollte diesen Code ausführen und zurückkehren.

Es wird empfohlen, alle diese Funktionsdeklarationen sowie den `CREATE LANGUAGE`-Befehl selbst in eine Erweiterung zu verpacken, sodass ein einfacher `CREATE EXTENSION`-Befehl ausreicht, um die Sprache zu installieren. Informationen zum Schreiben von Erweiterungen finden Sie in [Abschnitt 36.17](36_SQL_erweitern.md#3617-zusammengehörige-objekte-in-einer-erweiterung-paketieren).

Die in der Standarddistribution enthaltenen prozeduralen Sprachen sind gute Referenzen, wenn Sie einen eigenen Language Handler schreiben möchten. Sehen Sie im Unterverzeichnis `src/pl` des Quellbaums nach. Auch die Referenzseite zu `CREATE LANGUAGE` enthält nützliche Details.
